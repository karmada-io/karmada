# Karmada at a Glance

Karmada is a Kubernetes-native multi-cluster orchestrator. It sits in front of N member
clusters and propagates resources to them according to declarative policies. The control
plane runs its own apiserver (an aggregated apiserver, not a forked kube-apiserver) and
stores Karmada CRDs in its own etcd. Workloads applied to the Karmada control plane never
actually run there; they are projected onto member clusters via the pipeline below.

```
ResourceTemplate ──► PropagationPolicy ──► ResourceBinding ──► Work ──► member cluster apiserver
        (CRD-less, e.g. a Deployment)        (per-resource)     (per-cluster)
                                              + OverridePolicy applied here
```

Every term in that pipeline matters when an agent is asked to "create a policy" or
"debug why X did not propagate". The rest of this knowledge base unpacks each link.

## Core CRDs the skills must know

| CRD                       | Scope         | Short name | Purpose                                                            |
|---------------------------|---------------|------------|--------------------------------------------------------------------|
| PropagationPolicy         | Namespaced    | pp         | Select resources + pick target clusters                            |
| ClusterPropagationPolicy  | Cluster-wide  | cpp        | Same as above but matches resources across namespaces              |
| OverridePolicy            | Namespaced    | op         | Mutate fields on resources after they are bound to a cluster       |
| ClusterOverridePolicy     | Cluster-wide  | cop        | Cluster-scoped variant of OverridePolicy                           |
| ResourceBinding           | Namespaced    | rb         | Internal: the scheduling decision (one per resource template)      |
| ClusterResourceBinding    | Cluster-wide  | crb        | Internal: cluster-scoped binding                                   |
| Work                      | Namespaced    | -          | Internal: per-cluster manifest; one Work per (resource, cluster)   |
| Cluster                   | Cluster-wide  | -          | Member cluster registration in the control plane                   |

ResourceBinding and Work are produced by controllers, not by humans. Agents should
read them (for debug) but never generate them.

## Components in one paragraph each

- **karmada-apiserver** — aggregated apiserver. Stores Karmada CRDs. Users always talk
  to this, never to member apiservers directly.
- **karmada-controller-manager** — runs the binding controller, execution controller,
  namespace controller, taint controller, and more. It turns `ResourceTemplate +
  PropagationPolicy` into `ResourceBinding`, and `ResourceBinding` into `Work`.
- **karmada-scheduler** — turns a ResourceBinding's placement spec into a concrete list
  of target clusters and (for divided replicas) a per-cluster replica count.
- **karmada-webhook** — admission controller: validates PropagationPolicy and OverridePolicy
  on write, sets defaults, blocks dangerous mutations (e.g. immutable fields).
- **karmada-aggregated-apiserver** — proxies `/apis/cluster.karmada.io/v1alpha1/clusters/<x>/proxy`
  through to member clusters, so `kubectl --cluster=member1` works.
- **karmada-search** — optional component, separate aggregated apiserver, indexes resources
  from member clusters and offers `kubectl get pods -A` style cross-cluster queries.
- **karmada-descheduler** — periodically re-evaluates schedule decisions and triggers
  rebalancing when a member cluster's capacity changes.
- **karmada-scheduler-estimator** — runs in each member cluster, reports unschedulable
  pod count back to the scheduler so dynamic-weight policies can react to real capacity.

## The four canonical agent use cases

These are the four end-to-end scenarios every skill in this repository is evaluated against.
If a skill cannot serve at least one of them, it should not exist.

1. **Generate** — produce a syntactically valid PropagationPolicy + OverridePolicy from
   a workload manifest and a natural-language placement intent.
2. **Audit** — receive a policy YAML and return structured findings (errors, warnings,
   style suggestions) before the user applies it.
3. **Explain** — given a policy and a fleet, predict which clusters will receive replicas
   and why, including divided-replica math.
4. **Debug** — given symptoms ("Service X exists in the control plane but is missing in
   member2"), walk the pipeline and report the most likely failure point.

## Where to look in the source tree

- `pkg/apis/policy/v1alpha1/propagation_types.go` — PropagationPolicy + ClusterPropagationPolicy types
- `pkg/apis/policy/v1alpha1/override_types.go` — OverridePolicy + ClusterOverridePolicy types
- `pkg/apis/work/v1alpha2/binding_types.go` — ResourceBinding types
- `pkg/controllers/binding/binding_controller.go` — turns RB into Work
- `pkg/controllers/execution/execution_controller.go` — dispatches Work to member clusters
- `pkg/webhook/propagationpolicy/` — admission validation
- `charts/karmada/_crds/bases/policy/` — canonical CRDs (source of truth for schema files)

Skill files should cite these paths inline whenever they make a claim about behavior.
