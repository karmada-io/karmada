# Karmada Components

The components an agent must be able to name, place in the chain, and
inspect when something goes wrong.

## Host (control plane) components

| Component                       | Namespace        | Responsibility |
|---------------------------------|------------------|----------------|
| `karmada-apiserver`             | `karmada-system` | Serves the federated API (PropagationPolicy, OverridePolicy, ResourceBinding, Work, Cluster, ŌĆ”). Users `kubectl apply` here. |
| `karmada-controller-manager`    | `karmada-system` | Reconciles policies into ResourceBindings; owns most policy CRD logic. |
| `karmada-scheduler`             | `karmada-system` | Picks target clusters for each ResourceBinding. Writes `spec.clusters`. |
| `karmada-webhook`               | `karmada-system` | Admission validation/mutation for policy CRDs. Rejects malformed YAML up-front. |
| `karmada-aggregated-apiserver`  | `karmada-system` | Federated `proxy` endpoint for talking to member clusters through the host plane. |
| `karmada-search`                | `karmada-system` | Optional. Indexes member-cluster resources for cross-cluster search. |
| `etcd`                          | `karmada-system` | Backing store for the karmada-apiserver. |

## Member-cluster components

| Component        | Mode              | Responsibility |
|------------------|-------------------|----------------|
| `karmada-agent`  | pull              | Runs **inside** the member cluster, watches Works in `karmada-es-<cluster>` on the host plane, reconciles them into the member cluster. |
| (push)           | push              | The host-side execution controller writes directly to the member apiserver using credentials in a `Secret`. No member-side agent. |

`Cluster` resources have `spec.syncMode: Push | Pull`. That choice
changes which logs to inspect when sync fails.

## Object chain (one workload, three clusters)

```
User input on host:
  apps/v1 Deployment        nginx (ns=default)
  policy.karmada.io PropagationPolicy   nginx-global

Generated by karmada-controller-manager:
  ResourceBinding           nginx-deployment (ns=default)
    spec.clusters: [member-eu, member-cn]   ŌåÉ written by scheduler

Generated by execution controller, one per target:
  Work                      nginx-...-eu  (ns=karmada-es-member-eu)
  Work                      nginx-...-cn  (ns=karmada-es-member-cn)

Reconciled by karmada-agent or push executor:
  apps/v1 Deployment        nginx (in member cluster, ns=default)
```

## Execution namespace convention

For every cluster `<name>`, the host plane has a namespace
`karmada-es-<name>` containing the `Work` objects targeting it.
**Never** create user resources in this namespace ŌĆö it's owned by
Karmada's reconciler.

## Where to look when X is broken

| Symptom                           | First component to inspect          |
|-----------------------------------|-------------------------------------|
| Policy YAML rejected on apply     | `karmada-webhook` logs               |
| RB not created after apply        | `karmada-controller-manager` logs    |
| RB exists, no `spec.clusters`     | `karmada-scheduler` logs             |
| Work created, member resource missing | `karmada-agent` (pull) / push executor logs |
| Cross-cluster query fails         | `karmada-search` logs (if installed) |
